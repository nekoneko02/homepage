## 1. 画面構成

単一ページ（1ページ完結のランディング型）。README のページデザインに準拠。

```
┌─────────────────────────────────────┐
│ Header（ロゴ/名前・各SNSアイコンリンク）         │
├─────────────────────────────────────┤
│ Hero / Profile                      │
│   ヒーロー画像 ＋ 簡易プロフィール                │
│   （肩書き・一言・主要プラットフォームへの導線）        │
├─────────────────────────────────────┤
│ [カテゴリ選択バー]                          │
│   領域 / シリーズ / 種別 / プラットフォーム         │
│ ┌──────────┬──────────┬──────────┐     │
│ │ Content  │ Content  │ Content  │     │  ← コンテンツ一覧（カードグリッド）
│ │ Card     │ Card     │ Card     │     │
│ └──────────┴──────────┴──────────┘     │
├─────────────────────────────────────┤
│ Twitter（公式タイムライン埋め込み）              │
├─────────────────────────────────────┤
│ Footer（コピーライト・各リンク再掲）              │
└─────────────────────────────────────┘
```

## 2. セクション別仕様

### 2.1 Header
- 左: サイト名「ねこエンジニア」（クリックでトップへ）
- 右: 各プラットフォームへのアイコンリンク（Zenn / note / Booth / GitHub / Twitter）
- スクロール追従は任意（MVPでは固定なしでも可）。

### 2.2 Hero / Profile
- ヒーロー画像（`public/` 配置の静的画像、`next/image` で最適化）。
- プロフィール: 肩書き・活動領域（IT/数学/ボルダリング/プロダクト）・一言紹介。
- プロフィール文言は **repo 内設定（`data/profile.ts` または `profile.json`）** で管理（更新頻度低）。

### 2.3 カテゴリ選択バー（フィルタ）

フィルタ軸は4種:

| 軸 | 選択肢 | 付与方法 |
|---|---|---|
| **領域** | `IT` / `数学` / `ボルダリング` / `プロダクト` | topics/タグのキーワードマッチ（自動）|
| **シリーズ** | `ゼロから` など（動的に存在するものを列挙） | タイトル/タグのパターンマッチ（自動） |
| **種別** | `資格・勉強` / `単発` / `本` | 取得元・タグから判定（自動） |
| **プラットフォーム** | `Zenn` / `note` / `Booth` / `GitHub Pages` / `その他` | 取得元（固定） |

- 挙動:
  - 各軸内は単一選択（MVPは単一選択）、軸間は AND 条件。
  - 「すべて」リセットを用意。
  - フィルタ状態は URL クエリに反映（例 `?domain=IT&platform=zenn`）→ 共有・直リンク可能。
- 実装: 静的に全件を描画し**クライアント側でフィルタ**（SSGと両立、追加フェッチ不要）。

### 2.4 コンテンツ一覧（カードグリッド）
- 1カード = 1コンテンツ。新しい順（`publishedAt` 降順）で表示。
- カード表示項目:

  | 項目 | 必須 | 備考 |
  |---|---|---|
  | タイトル | ✅ | |
  | プラットフォーム識別（バッジ/アイコン） | ✅ | Zenn/note/Booth/GitHub/Product |
  | 公開日 | ✅ | 取得できる場合 |
  | サムネイル | 任意 | 無い場合はプラットフォーム別プレースホルダ |
  | 抜粋 | 任意 | RSS等から取得できる場合 |
  | カテゴリタグ（領域） | 任意 | 自動付与されたもの |

- クリックで **元プラットフォームを別タブで開く**（`target="_blank" rel="noopener"`）。
- 0件時: 「該当するコンテンツがありません」を表示。
- レスポンシブ: モバイル1列 / タブレット2列 / デスクトップ3列。

### 2.5 Twitter 埋め込み
- 公式の埋め込みタイムライン（widgets.js）を使用。
- スクリプトはクライアントでのみ読み込み（`next/script`、`strategy="lazyOnload"`）。
- 読み込み失敗・ブロック時もレイアウトが崩れないようフォールバック高さを確保。

### 2.6 Footer
- コピーライト、各プラットフォームリンク再掲、最終ビルド日時（任意）。

### 2.7 Google Analytics
- GA4 を使用。
- **計測必須**: ページビュー（`page_view`）。
- **計測推奨（必要に応じて）**: スクロール率、外部リンククリック（各プラットフォームへの遷移）。
- `next/script`（`strategy="afterInteractive"`）で `gtag.js` を読み込む。
- 計測 ID（`G-XXXXXXXXXX`）は環境変数 `NEXT_PUBLIC_GA_MEASUREMENT_ID` で管理。

## 3. データ取得タイミング仕様

- すべて **ビルド時（SSG）に確定**。閲覧時は静的HTML/JSONのみ。
- 自動取得対象（Zenn/note/GitHub）: ビルド時にフェッチ → 正規化 → カテゴリ付与 → 集約。
- 手動対象（Booth/個人開発/プロフィール）: repo 内データを読み込み。
- 再ビルドはスケジュール実行（[`03_design.md` §6](./03_design.md#6-デプロイと再ビルド)）。
- 取得失敗時はスナップショットへフォールバック（FR-09）。

## 4. カテゴリ自動付与 仕様（FR-08）

**方針**: プラットフォームが持つメタデータを直接利用する。手動マッピング表は設けない。

| ソース | 使うデータ | 判定優先度 |
|---|---|---|
| Zenn | `type: "tech"` → IT確定 / `topics` の値をキーワードマッチ | 最優先 |
| note | タイトルへのキーワードマッチ | フォールバック |
| GitHub | `topics` の値をキーワードマッチ | 最優先 |
| 手動JSON | `tags` の値をキーワードマッチ | 最優先 |

**キーワードマッチの方式**（シンプルに保つ）:

```
領域 = 次のキーワードがタグかタイトルに含まれるか:
  IT       : typescript, javascript, python, aws, gcp, react, next.js, docker, linux,
             プログラミング, エンジニア, 開発, API, ...
  数学     : 数学, math, 線形代数, 微積分, 確率, 統計, ...
  ボルダリング: ボルダリング, クライミング, boulder, climb, ...
  プロダクト : プロダクト, サービス, リリース, startup, ...

シリーズ = タイトルが次のパターンに一致するか:
  "ゼロから" : /ゼロから/
  （今後追加可能）

種別 = 次のルールで判定:
  本         : GitHub Pages 由来 or Booth 由来 or タグに "本"
  資格・勉強 : タイトルかタグに /資格|試験|勉強|検定/
  単発       : 上記以外
```

- **マッチしない場合**: 領域はプラットフォームの既定値（Zenn→IT、GitHub→数学 等）、種別は「単発」。
- **キーワードリストは設定ファイルに集約**し、追加は1行で済む（FR-08 実装要件）。
- Zenn の `type: "tech"` のように**プラットフォーム公式の分類を最優先**し、キーワードマッチはその補助。

## 5. SEO / メタ仕様
- `<title>`, `<meta description>`, OGP（og:title/description/image）, Twitter Card。
- `sitemap.xml` / `robots.txt` を出力。
- 構造化データ（JSON-LD `Person`）でプロフィールを記述（任意）。

## 6. エラー/空状態の仕様

| 状況 | 挙動 |
|---|---|
| 特定ソースの取得失敗 | 当該ソースはスナップショット値で表示、他は通常表示 |
| 全ソース取得失敗 | スナップショット全体で表示。ビルドは成功させる |
| Twitter 読み込み失敗 | 当該セクションのみ非表示/プレースホルダ |
| フィルタ結果0件 | 空メッセージ表示 |

## 7. 手動更新の仕様（NFR-02）

手動更新が必要なコンテンツ（Booth / 個人開発 / 数学本〔MVP期〕）の運用手順:

1. `data/booth.json`（or `products.json` / `books.json`）を開く。
2. 最後の要素の後ろに1オブジェクトを追記する。
3. `git commit & push` → 自動デプロイ。

必要フィールド（最小）:

```jsonc
{
  "title": "タイトル",         // 必須
  "url": "https://...",        // 必須（遷移先URL）
  "publishedAt": "2025-01-01", // 任意（降順ソートに使用）
  "tags": ["数学", "本"],      // 任意（カテゴリ自動判定に使用）
  "thumbnail": "https://..."   // 任意
}
```

**バリデーション**: ビルド時に JSON スキーマチェックを行い、不正データはビルドエラーとして検出する（サイレントスキップしない）。
