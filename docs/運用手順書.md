# 運用手順書 — ねこエンジニア ホームページ

## 概要

本ホームページは **「できる限り更新しなくてよい」** 設計です。

| コンテンツ | 更新方法 | 頻度 |
|---|---|---|
| Zenn 記事・本 | 手動（Zennリポジトリの feed.json に追記） | 投稿時のみ |
| note 記事 | 自動（毎日 21:00 JST 再ビルド） | 放置でOK |
| GitHub リポジトリ（数学本・個人開発など） | 自動（GitHub Topic + feed.json） | 放置でOK |
| Booth 同人誌 | 手動（JSON 1行追記） | 出版時のみ |

## GitHub リポジトリをホームページに掲載する

リポジトリに **`neko-homepage` Topic** を付けるだけで、次回ビルド時に自動掲載されます。

### 基本手順（Topic を付けるだけ）

1. GitHub でリポジトリを開く
2. 右上の歯車アイコン（About）→ Topics に `neko-homepage` を追加 → **Save changes**
3. 翌日 21:00 JST の自動ビルドで反映（今すぐ反映したい場合は後述の手動トリガーを使用）

feed.json がない場合は、リポジトリの **名前・説明（Description）・Website URL** がそのまま使われます。

プラットフォーム（オリジナルサイト扱いにするか）は Topics で制御できます：

| Topics に追加 | プラットフォーム | 自動付与される領域 |
|---|---|---|
| `neko-homepage` のみ | GitHub | その他（またはキーワード判定） |
| `neko-homepage` + `site` | オリジナルサイト | プロダクト |

### 詳細なコンテンツ情報を載せる（feed.json）

`publishedAt`・説明文・プラットフォームを指定したい場合は、リポジトリのルートに `feed.json` を置きます。

#### パターン①：リポジトリ全体を1件として掲載（プロダクト・本1冊）

```json
{
  "platform": "site",
  "items": [
    {
      "id": "my-product",
      "title": "プロダクト名",
      "url": "https://my-product.com",
      "date_published": "2025-06-01",
      "summary": "説明文",
      "tags": ["プロダクト", "個人開発"]
    }
  ]
}
```

#### パターン②：章単位・エントリ単位で複数掲載（連載本など）

```json
{
  "platform": "github",
  "items": [
    {
      "id": "chapter-1",
      "title": "第1章 ベクトル空間",
      "url": "https://nekoneko02.github.io/linear-algebra/chapter1",
      "date_published": "2025-01-01",
      "tags": ["数学", "線形代数", "本"]
    },
    {
      "id": "chapter-2",
      "title": "第2章 線形写像",
      "url": "https://nekoneko02.github.io/linear-algebra/chapter2",
      "date_published": "2025-02-01",
      "tags": ["数学", "線形代数", "本"]
    }
  ]
}
```

| フィールド | 説明 |
|---|---|
| `platform` | `"github"`（数学本など）または `"site"`（個人開発プロダクト）。省略時は `"github"` |
| `items[].id` | 重複しない任意の文字列 |
| `items[].date_published` | 公開日（降順ソートに使用） |
| `items[].summary` | 説明文 |
| `items[].thumbnail` | サムネイル画像URL（省略時はプラットフォームカラーで自動生成） |
| `items[].tags` | カテゴリ自動判定に使用。省略時はリポジトリの Topics を使用 |
| `items[].visibility` | 表示先の配列。省略時は `["content"]`。詳細は後述 |
| `items[].news_text` | NEWSセクション用の短い説明文（`visibility` に `"news"` を含む場合のみ表示） |

## 手動コンテンツ追加

### Zenn 記事・本を追加する

`nekoneko02/zenn` リポジトリのルートにある `feed.json` に追記して push します。

```jsonc
// feed.json（抜粋）
{
  "platform": "zenn",
  "items": [
    {
      "id": "記事スラッグ（URLの末尾）",
      "title": "記事タイトル",
      "url": "https://zenn.dev/neko_student/articles/記事スラッグ",
      "date_published": "2025-01-01",        // 公開日（降順ソートに使用）
      "tags": ["AWS", "初心者"],
      "thumbnail": "https://...",            // 任意: サムネイル画像URL
      "visibility": ["content", "news"],     // 任意: 表示先。省略時は ["content"]
      "news_text": "新記事を公開しました！"  // 任意: NEWSセクション用の短い説明文
    }
  ]
}
```

Zenn Bookの場合はURLを `books/スラッグ` に変え、`"tags"` に `"book"` を含めると種別「本」として分類されます。

`visibility` と `news_text` の詳細は「[NEWSセクションに掲載する](#newsセクションに掲載する)」を参照してください。

---

### NEWSセクションに掲載する

トップページの Hero 直下に横スクロールで表示される「NEWS」セクションへの掲載は、`visibility` フィールドで制御します。

| `visibility` の値 | コンテンツ一覧 | NEWSセクション | 用途例 |
|---|---|---|---|
| `["content"]`（省略時） | ✅ | — | 通常のコンテンツ |
| `["content", "news"]` | ✅ | ✅ | 新刊リリースなど |
| `["news"]` | — | ✅ | 新章追加・お知らせなど |

`["news"]` のみにすると、コンテンツ一覧には出ず NEWS セクションだけに表示されます。同じ本に章を追加した際のお知らせなど、重複を避けたい場合に使います。

`newsText` を指定すると、NEWS カードにタイトルとは別の短い説明文（改行対応）を表示できます。

### Booth 同人誌を追加する

`data/booth.json` に1オブジェクトを追記して `git push` するだけです。

```jsonc
// data/booth.json
[
  {
    "title": "ゼロから学ぶ〇〇",           // 必須: タイトル
    "url": "https://neko-engineer.booth.pm/items/xxxx",  // 必須: Booth の商品URL
    "publishedAt": "2025-01-01",           // 任意: YYYY-MM-DD（降順ソートに使用）
    "tags": ["数学", "本"],                // 任意: カテゴリ自動判定に使用
    "thumbnail": "https://...",            // 任意: サムネイル画像URL
    "excerpt": "〇〇についての入門書です。", // 任意: 説明文
    "visibility": ["content", "news"],     // 任意: 表示先。省略時は ["content"]
    "newsText": "新刊リリースしました！"    // 任意: NEWSセクション用の短い説明文
  }
]
```

**手順:**
1. `data/booth.json` を開く
2. 配列の末尾に上記フォーマットでオブジェクトを追記
3. `git commit -m "add: Booth 同人誌「タイトル」を追加" && git push`
4. Amplify が自動デプロイ（push トリガー）

## カテゴリ（領域）の自動判定ルール

### 領域の値一覧

| 領域 | 説明 |
|---|---|
| `IT` | プログラミング・技術系 |
| `数学` | 数学・統計系 |
| `ボルダリング` | ボルダリング・クライミング系 |
| `プロダクト` | 個人開発・サービス系 |
| `ブログ` | ブログ記事全般 |
| `本` | 書籍・同人誌 |
| `その他` | 上記に当てはまらないもの |

### 判定の優先順位

1. **プラットフォーム固定** — プラットフォームに応じて常に付与される領域

   | プラットフォーム | 常に付与される領域 |
   |---|---|
   | Zenn | IT、ブログ |
   | note | ブログ |
   | Booth | 本 |
   | オリジナルサイト（site） | プロダクト |

2. **キーワードマッチ** — タイトルや tags に以下のキーワードが含まれると自動付与

   | 領域 | 代表キーワード |
   |---|---|
   | IT | typescript, python, aws, react, docker, プログラミング, 開発 ... |
   | 数学 | 数学, math, 線形代数, 確率, 統計 ... |
   | ボルダリング | ボルダリング, クライミング, boulder ... |
   | プロダクト | プロダクト, サービス, リリース, startup ... |

3. **デフォルト** — 上記いずれにも該当しない場合は「その他」

### feed.json の tags に書く値の目安

GitHub リポジトリの `feed.json` で tags を指定すると、キーワードマッチに使われます。

```json
"tags": ["数学", "線形代数", "本"]
```

| 付けたい領域 | tags に含めるとよいキーワード |
|---|---|
| 数学 | `数学`, `math`, `線形代数` など |
| IT | `プログラミング`, `typescript`, `aws` など |
| ボルダリング | `ボルダリング`, `クライミング` |
| プロダクト | `プロダクト`, `サービス` |
| 本 | `本`, `book` |

キーワードを追加・修正したい場合は `content/category-rules.ts` を編集してください。

## 環境変数の管理

### ローカル開発

`.env.local.example` をコピーして `.env.local` を作成します。

```bash
cp .env.local.example .env.local
```

`.env.local` は `.gitignore` に含まれているため、リポジトリにはコミットされません。

### Amplify Hosting（本番）

Amplify Console → App → **Environment variables** で設定します。

| 変数 | 設定値 | 必須 |
|---|---|---|
| `NOTE_RSS_URL` | `https://note.com/alg_geo/rss` | ✅ |
| `NEXT_PUBLIC_GA_MEASUREMENT_ID` | `G-XXXXXXXXXX` | ✅（本番） |
| `GITHUB_TOKEN` | GitHub PAT（`read:public_repo` スコープ） | 推奨 |
| `GITHUB_USER` | `nekoneko02`（GitHub ユーザー名、省略時はデフォルト値を使用） | 任意 |

## 付録：自動更新の仕組み

### 仕組み

GitHub Actions が毎日 **21:00 JST**（UTC 12:00）に起動し、  
note・GitHub リポジトリの最新コンテンツを取得して差分があれば Amplify 再ビルドを Webhook でトリガーします。

```
スケジューラ（GitHub Actions cron）
  → 差分チェック（scripts/check-content-diff.js）
  → 差分あり → Amplify Webhook → 再ビルド → 自動デプロイ
```

### やること（定常）

**Zenn は手動更新が必要です。** Zennに記事・本を公開したら `nekoneko02/zenn` リポジトリの `feed.json` に追記して push してください。note は放置でOK。GitHub リポジトリは `neko-homepage` Topic を付けるだけで翌日 21:00 JST 以降に自動反映されます。

### 手動で今すぐ反映したいとき

GitHub Actions の **workflow_dispatch** から手動実行します。

1. GitHub リポジトリ → **Actions** タブ
2. 左ペイン: **Scheduled Rebuild** を選択
3. **Run workflow** ボタン → **Run workflow** を実行

### GitHub Actions に必要な Secrets

Amplify Console の「Webhook」URLを取得し、GitHub Secrets に登録してください。

| Secret 名 | 値 | 取得場所 |
|---|---|---|
| `AMPLIFY_BUILD_WEBHOOK` | `https://webhooks.amplify.ap-northeast-1.amazonaws.com/prod/webhooks?id=...` | Amplify Console → App → Webhooks |
