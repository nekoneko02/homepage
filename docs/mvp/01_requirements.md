# 要件定義書 — ねこエンジニア ホームページ

> 本書は `README.md`（背景・目的・方針）を上位文書とし、それを要件として具体化したものです。
> 仕様は [`02_specification.md`](./02_specification.md)、設計は [`03_design.md`](./03_design.md)、
> 各プラットフォームの取得方針は [`04_content-sources.md`](./04_content-sources.md) を参照。

## 1. 目的（再掲・確認）

- **人物紹介**: 「ねこエンジニアとはこういう人物だ」が一目でわかる。
- **ハブ機能**: 分散したコンテンツ（Zenn / note / Booth / Twitter / GitHub Pages / 個人開発）の導線を集約し、
  プラットフォーム横断の回遊（例: Zenn → note）を実現する。

## 2. スコープ

### 2.1 やること（In Scope）
- トップ（ヒーロー＋簡易プロフィール）
- 全コンテンツを横断する一覧表示
- カテゴリによる絞り込み（領域 / シリーズ / 種別 / **プラットフォーム**）
- 各プラットフォームへの外部リンク導線
- Twitter 公式埋め込み
- Google Analytics によるアクセス計測

### 2.2 やらないこと（Out of Scope）
- コンテンツ本文の保持・配信（**本体はハブに徹し、中身は各プラットフォームを正とする**）
- ログイン / 会員機能 / コメント / 問い合わせフォーム（初期は不要）
- 多言語対応（日本語のみ）
- CMS の導入（更新は「ソース＝各プラットフォーム」または最小限の repo 内設定で行う）

## 3. ユーザーと利用シナリオ

| ペルソナ | 来訪経路 | 達成したいこと |
|---|---|---|
| Zenn 読者 | Zenn記事のプロフィールリンク | 他の活動（note/本/プロダクト）を知り回遊する |
| note 読者 | note プロフィール | 技術面の活動（Zenn等）を知る |
| 初見の人 | SNS / 検索 | 「何をやっている人か」を短時間で把握する |
| 採用・依頼者 | 名刺 / 紹介 | 経歴と成果物の所在を確認する |

## 4. 機能要件（FR）

| ID | 要件 | 優先度 |
|---|---|---|
| FR-01 | トップにヒーロー画像と簡易プロフィール（肩書き・活動領域・各SNSリンク）を表示する | 必須 |
| FR-02 | 全プラットフォームのコンテンツを統一フォーマットの一覧で表示する | 必須 |
| FR-03 | 各コンテンツカードから該当プラットフォームの元ページへ遷移できる | 必須 |
| FR-04 | カテゴリ（**領域 / シリーズ / 種別 / プラットフォーム**）で一覧を絞り込める | 必須 |
| FR-05 | 自動取得対象（Zenn/note 等）はビルド時に最新化される | 必須 |
| FR-06 | 自動取得が困難なもの（Booth等）は repo 内データで手動更新でき、手順が簡潔であること | 必須 |
| FR-07 | Twitter のタイムラインを公式埋め込みで表示する | 必須 |
| FR-08 | カテゴリは**プラットフォームのメタデータ（topics/タグ）を直接利用して自動付与**し、判定不能な場合のみタイトルキーワードマッチを補助的に使う。手動マッピングは設けない | 必須 |
| FR-09 | 外部取得が失敗してもサイトはビルド・表示できる（前回スナップショットにフォールバック） | 推奨 |
| FR-10 | Google Analytics を組み込み、**最低限ページビュー**を計測する | 必須 |
| FR-11 | **コンテンツに差分がない場合は Amplify 再ビルドをスキップ**できる（オプション・実装容易な場合） | 推奨 |
| FR-12 | ビルドをローカル環境または GitHub Actions 手動実行でいつでも実行できる | 必須 |

## 5. 非機能要件（NFR）

| ID | 区分 | 要件 |
|---|---|---|
| NFR-01 | 運用（自動） | **できる限り更新作業が不要**であること。定常運用＝放置でも内容が最新化される。 |
| NFR-02 | 運用（手動） | **手動更新が必要なコンテンツ（Booth等）は、JSONファイルに1行追加するだけで反映できる**シンプルな運用手順であること。 |
| NFR-03 | 安定性 | **非公式API・スクレイピングは原則不使用**。公式フィード/公式APIを優先する。 |
| NFR-04 | コスト | 個人運用。AWS Amplify の無料〜低額枠に収まる構成とする。 |
| NFR-05 | 可用性 | 外部プラットフォーム障害時もホームページ自体は閲覧可能（FR-09）。 |
| NFR-06 | 保守性 | プラットフォーム追加が「取得アダプタ1つの追加」で済む拡張構造とする。 |
| NFR-07 | アクセシビリティ | レスポンシブ対応、主要要素にセマンティックHTML/altを付与。 |
| NFR-08 | SEO | 各種メタタグ・OGP・サイトマップを出力する。 |
| NFR-09 | 計測 | Google Analytics（最低限 PV 計測、必要に応じて追加メトリクスを拡張可能な設計とする）。 |

## 6. 制約・前提

- 技術スタック: **Next.js（App Router） + AWS Amplify Hosting**。
- レンダリング: **SSG（ビルド時取得）＋定期再ビルド**（ISR/SSRは使わない＝静的配信で完結）。
- データの「正」は各プラットフォーム側。ホームページは取得・整形・集約のみ。
- 更新頻度が極端に低いもので GitHubリポジトリを持たないものは手動更新を許容。
- **GitHubリポジトリが存在するものは、自前の構造化データ（frontmatter / Topics）を活用した自動取得を優先する**。

## 7. GitHub リポジトリ全自動取得 方針（FR への追加検討）

GitHubリポジトリを持つコンテンツ（数学本・個人開発プロダクト等）について、長期的に以下の方針を採用する。

**基本方針**: 対象リポジトリに **専用Topic（`neko-homepage`）** を付与することで、
`GET /users/nekoneko02/repos?topic=neko-homepage` 一発でホームページ掲載対象を全自動取得する。

- 追加は「リポジトリに `neko-homepage` トピックを付ける」だけ。コード変更不要。
- リポジトリの `description`, `homepage`, `topics`, `language` を取得してメタデータ化。
- カテゴリはリポジトリの topics から自動判定（FR-08 と同一フレーム）。

**MVPのスコープ**: 方針確定のみ。実装は MVP 後の段階導入とする。
MVP では数学本・個人開発は `data/books.json` / `data/products.json` の手動枠で開始し、
実装後に自動取得に切り替える（データ構造は共通化するため切替コストは小さい）。

## 8. 受け入れ基準（MVP）

- [ ] トップでプロフィールと全SNS導線が表示される。
- [ ] Zenn・note のコンテンツがビルド時に自動取得され一覧表示される。
- [ ] カテゴリ絞り込みが**領域・プラットフォーム**単位で機能する。
- [ ] Twitter 埋め込みが表示される。
- [ ] Booth / GitHub Pages / 個人開発は手動データで表示できる（自動化は段階導入）。
- [ ] スケジュール再ビルドにより、手作業なしで Zenn/note の新着が反映される。
- [ ] `npm run build` でローカルビルドが成功し、成果物が確認できる。
- [ ] GitHub Actions の手動実行（workflow_dispatch）でいつでも再ビルドできる。
- [ ] Google Analytics のページビューが計測されている（本番環境確認）。
- [ ] Booth を `data/booth.json` に1行追記するだけで一覧に反映される。
